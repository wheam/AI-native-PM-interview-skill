# 校准锚点样本：3.0 答卷长什么样

> 用途：给"校准锚点=3.0"一个**真实可对照的样本**。打分时拿候选人表现和这份比，不要凭空估"3.0 是什么感觉"。
> 来源：本 skill 作者 2026-06 第一次完整 10 题模拟面试。旧标准误判为 3.5，复盘从严校准锚定 3.0。

## 这份答卷的画像（= 3.0 的典型形态）

**强项**：应用层模型理解扎实、一线手感真（claude code harness 演变、hermes 多槽位降级、文生图→图生视频等真实案例密度高）、Product Sense Case 有创造力、行业判断独到、Builder 有 GitHub 全绿硬证据。

**致命短板**：
1. **底层机制薄** — 理解停在应用层，到不了机制级（attention/KV cache/MoE/微调 vs RAG 的机制切分）。典型暴露：把 fine-tune vs RAG 切成"同类问题 vs 通用问题"，而非准确的"教行为/风格 vs 注入知识"。
2. **传统 PM 强、AI native 不足** — 经典题（如 DAU 涨留存跌）用纯传统 UG 拆解框架答，零 AI 视角（没提生成质量波动、模型版本切换、token 成本与留存关系）。
3. **缺量化标杆战绩** — 有真实项目但全程无"X 做到 Y 规模/+Z%"硬数字。

## 旧标准 → 新标尺的分项平移（关键示范）

| 能力 | 权重 | 旧分 | 新分 | 平移理由 |
|------|------|------|------|----------|
| 模型 Sense | 25% | 3.5 | **2.5** | 机制层的洞，候选人自己都点出来了 |
| Eval | 15% | 4.0 | **3.5** | 仍是最亮，但**吃了开卷长答的便宜**，闭卷口头未必撑住 4.0 |
| Product Sense | 20% | 4.0 | **3.5** | 同上，Case 强但开卷红利，闭卷追问下降档 |
| Builder | 15% | 4.0 | **3.0** | GitHub 全绿证明高频用，但证明不了产品战绩，缺量化标杆 |
| 数据&商业化 | 15% | 3.5 | **2.5** | 一道纯传统 UG、不 AI 的答案直接拖下来 |
| 跨职能&Safety | 10% | 3.5 | **3.0** | 扎实有监管意识，但深度一般 |

**加权 ≈ 2.98 → 锚定 3.0。**

> **注：本表"新分"按 `scoring-rubric.md` 第二节的新锚点矩阵读取**（4=该项行业顶尖 / 3.5=非常优秀 / 3=扎实合格但不突出 / 2=明显不足）。
> - 本样本最高项是 3.5（Eval、Product Sense），**没有任何一项摸到 4（顶尖）**——这正是后来"把维度 4 重定义为行业顶尖、原 4 文本降为 3.5"对本锚点零影响的原因，加权仍 ≈3.0。
> - Eval / Product 的 3.5 是**上界判断**：开卷长答里像 4，但去掉开卷红利、闭卷口头追问后落在"非常优秀"(3.5) 这一档，不到"顶尖"(4)。这两项也是后续若补 3.5 锚点样本时最该拿来对照的。

## 从这份样本提炼的打分纪律（已并入 SKILL.md，此处给"为什么"）

1. **不能只压一项**：复盘时若只把模型 sense 降到 2.5、其余仍挂 4.0/3.5，加权会飘回 3.5。**各单项必须跟着标尺一起从严平移**，否则锚点形同虚设。
2. **开卷红利要扣回**：批量笔答下拿到的 4.0（Eval/Product 这种能慢慢铺长答的题）含水分。闭卷口头追问场景下，默认这类高分守不住，降半档是合理的。
3. **两个 2.5 拖一堆 3.5 = 3.0**：这是 3.0 的典型加权结构。涨分路径很明确——把拖后腿的 2.5 补成 3.5，总分自然上 3.5。补的只能是 AI 侧（机制理解、AI 视角答经典题），传统 PM 话术刷不上去。

## 怎么用这份锚点

- 给新候选人打分时，先问"他比这份样本强在哪、弱在哪"。
- 比样本明显强（机制层能对话、有量化标杆、闭卷追问不崩）→ 才够 3.5+。
- 和样本同档（应用层强、机制薄、传统 PM 撑场）→ 就是 3.0，别因为某几题亮就给 3.5。
- 比样本弱（应用层都讲不透、纯传统视角）→ 往 2.5 走。
