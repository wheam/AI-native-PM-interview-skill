# 校准锚点样本：3.0 答卷长什么样

> 用途：给"校准锚点=3.0"一个**真实可对照的样本**。打分时拿候选人表现和这份比，不要凭空估"3.0 是什么感觉"。
> 来源：本 skill 作者 2026-06 第一次完整 10 题模拟面试。旧标准误判为 3.5，复盘从严校准锚定 3.0。

## 这份答卷的画像（= 3.0 的典型形态）

**强项**：应用层模型理解扎实、一线手感真（能讲清 coding agent 的 harness 随模型代际演变、某 agent 产品的多模型槽位降级、文生图→图生视频等真实案例密度高）、Product Sense Case 有创造力、行业判断独到、Builder 有 GitHub 全绿硬证据。

**致命短板**：
1. **底层机制薄** — 理解停在应用层，到不了机制级（attention/KV cache/MoE/微调 vs RAG 的机制切分）。典型暴露：把 fine-tune vs RAG 切成"同类问题 vs 通用问题"，而非准确的"教行为/风格 vs 注入知识"。
2. **传统 PM 强、AI native 不足** — 经典题（如 DAU 涨留存跌）用纯传统 UG 拆解框架答，零 AI 视角（没提生成质量波动、模型版本切换、token 成本与留存关系）。
3. **缺量化标杆战绩** — 有真实项目但全程无"X 做到 Y 规模/+Z%"硬数字。

## 旧标准 → 新标尺的分项平移（关键示范）

| 能力 | 权重 | 旧分 | 新分 | 平移理由 |
|------|------|------|------|----------|
| 模型 Sense | 25% | 3.5 | **2.5** | 机制层的洞，候选人自己都点出来了 |
| Eval | 15% | 4.0 | **3.5** | 仍是最亮，但**吃了开卷长答的便宜**，闭卷口头未必撑住 4.0 |
| Product Sense | 20% | 4.0 | **3.5** | 同上，Case 强但开卷红利，闭卷追问下降档 |
| Builder | 15% | 4.0 | **3.0** | GitHub 全绿证明高频用，但证明不了产品战绩，缺量化标杆 |
| 数据&商业化 | 15% | 3.5 | **2.5** | 一道纯传统 UG、不 AI 的答案直接拖下来 |
| 跨职能&Safety | 10% | 3.5 | **3.0** | 扎实有监管意识，但深度一般 |

**加权 ≈ 2.98 → 锚定 3.0。**

> **注：本表"新分"按 `scoring-rubric.md` 第二节的新锚点矩阵读取**（4=该项行业顶尖 / 3.5=非常优秀 / 3=扎实合格但不突出 / 2=明显不足）。
> - 本样本最高项是 3.5（Eval、Product Sense），**没有任何一项摸到 4（顶尖）**——这正是后来"把维度 4 重定义为行业顶尖、原 4 文本降为 3.5"对本锚点零影响的原因，加权仍 ≈3.0。
> - Eval / Product 的 3.5 是**上界判断**：开卷长答里像 4，但去掉开卷红利、闭卷口头追问后落在"非常优秀"(3.5) 这一档，不到"顶尖"(4)。这两项也是后续若补 3.5 锚点样本时最该拿来对照的。

## 证据制严判（v1.1 新标准 · 逐字稿对照 · 2026-06-06）

> 用 v1.1 六步流程（证据取原话 → 先列红旗 → Bar-raiser 挑战 ≥3.5）对同一份逐字稿重判一遍。注意：原答卷是**开卷批量笔答**，「开卷红利要扣回」全程适用。逐字稿由维护者本地留存，不随 skill 分发。

| 能力 | 权重 | v1.1 严判 | 关键证据 / 红旗 |
|------|------|-----------|----------------|
| 模型 Sense | 25% | **2.5** | ⑤harness 演变、⑧记忆系统手感真（不低于 2.5）；但 ②"同类→fine-tune/通用→RAG"是**核心概念讲反** + 零机制层 → 压回 2.5 |
| Eval | 15% | **3.0**（↓） | ③ 全场最成熟，但是"**出色的指标设计**"，**不是新锚点要的 trace 级 eval 手艺**（无 error analysis / trajectory / judge craft）→ 够不到新 3.5 |
| Product Sense | 20% | **3.5** | ①④⑩ 三个独立强证据三角互证，好在"看得见的推理"非"无法验证的自述"，**Bar-raiser 没挑下去** → 真实峰值，顶住 3.5 |
| Builder | 15% | **3.0** | ⑦ GitHub 全绿 + 真实在建产品 + pipeline 反思；但无上线战绩/用户规模 → 强 3.0 |
| 数据&商业化 | 15% | **2.5** | ⑩ 商业化 sense 强；但 ⑥ 经典题**纯传统 UG 零 AI 视角** + 全场零量化战绩（撞自述封顶）→ 数据半边塌 |
| 跨职能&Safety | 10% | **3.0** | ⑨ 监管意识扎实；无 research partnership、深度一般 |

**加权 ≈ 2.90 → 仍锚定 3.0。**（盲估画像时 2.80，逐字稿证据制后 2.90，旧标准 2.98——三者收敛在 3.0。）

**两条最该记住的判分先例（将来给别人打分直接可引）：**
1. **③ 指标设计 ≠ eval 手艺**：能列出一套漂亮的线上线下指标，是 3.0；新 3.5/4 要的是 trace 级 error analysis → failure taxonomy → trajectory/judge craft。这是 v1.1 收紧 Eval 锚点咬到的地方。
2. **② fine-tune/RAG 讲反**：把切分说成"同类问题 vs 通用问题"（正确是 fine-tune 教行为/风格、RAG 注入知识），在 25% 权重维度上直接吃掉半档——核心概念错误比"没覆盖"更扣分。

## 从这份样本提炼的打分纪律（已并入 SKILL.md，此处给"为什么"）

1. **不能只压一项**：复盘时若只把模型 sense 降到 2.5、其余仍挂 4.0/3.5，加权会飘回 3.5。**各单项必须跟着标尺一起从严平移**，否则锚点形同虚设。
2. **开卷红利要扣回**：批量笔答下拿到的 4.0（Eval/Product 这种能慢慢铺长答的题）含水分。闭卷口头追问场景下，默认这类高分守不住，降半档是合理的。
3. **两个 2.5 拖一堆 3.5 = 3.0**：这是 3.0 的典型加权结构。涨分路径很明确——把拖后腿的 2.5 补成 3.5，总分自然上 3.5。补的只能是 AI 侧（机制理解、AI 视角答经典题），传统 PM 话术刷不上去。

## 怎么用这份锚点

- 给新候选人打分时，先问"他比这份样本强在哪、弱在哪"。
- 比样本明显强（机制层能对话、有量化标杆、闭卷追问不崩）→ 才够 3.5+。
- 和样本同档（应用层强、机制薄、传统 PM 撑场）→ 就是 3.0，别因为某几题亮就给 3.5。
- 比样本弱（应用层都讲不透、纯传统视角）→ 往 2.5 走。

# 第二个 3.0 样本（脱敏 · 镜像互补型）

> 用途：3.0 这一档原本只有上面一个样本（"builder / 产品愿景型"）。再加一个**强弱项几乎镜像**的样本，把 3.0 从"一个点"撑成"一条线"——打分时两份都拿来对照，更稳。
> 已脱敏：隐去姓名与逐字稿，只保留画像与评分依据类型。

## 画像（= 另一种 3.0 形态："数据 / 评测型"）

**背景**：大厂 本地生活 / 地图类「图像质量评估 + 数据产品 + 商业化策略」PM；研究生 计算数学 + 图像方向。
**强项**：Eval 是真手艺（不是会列指标，是真在做 diff-case → 沉淀 policy → 转 prompt 约束 → 重跑 的迭代闭环、severity 分层单独评估、人工 + LLM 标注融合，还能主动指出"蒸馏小模型会学到 SOTA 偏见而非真实用户偏好"）；模型概念无硬错（RAG 注入知识 / SFT 管风格格式 / Prompt 管临时变量，切分讲对）；真·数据产品域深度。
**致命短板**：
1. **Builder 弱** — 全是个人提效（取数 chatbot 等），无真实用户 / 无量化 / 还有"100% 成功率"式夸大 / 自己的工具不做 eval。
2. **Product Sense 缺创造性与战略** — 能拆体验 vs 业务指标、case 归因，但指标组合 + 灰度/回滚说不硬，没有 standalone 设计题里的独到判断。
3. **经典 metrics 题仍漏 AI 视角** — 成本/质量题里一开始用"复访/交互深度/点赞率"衡量体验（engagement 残留），和样本 A 的"DAU 涨留存跌纯传统拆解"是同一类毛病。

## 六维评分（v1.1 证据制，依据类型而非原话）

| 能力 | 权重 | 评分 | 依据 |
|------|------|------|------|
| 模型 Sense | 25% | **3.0** | RAG/FT 概念切分正确、懂多模态 LLM vs 视觉分类 + 蒸馏训练数据；但机制层"偏应用直觉"、未被硬探 |
| Eval | 15% | **3.5** | diff-case→policy→重跑的真手艺 + severity 分层 + 主动提偏见风险；线上验证偏弱、单一领域，故未到 4 |
| Product Sense | 20% | **3.0** | 体验/业务指标拆分、case 归因到位；指标组合 + 灰度/回滚没正面答、缺创造性 |
| Builder | 15% | **2.5** | 纯个人提效、无用户无量化、有夸大、自工具不做 eval |
| 数据&商业化 | 15% | **3.0** | 真数据产品域深度；但 engagement 指标残留、无量化战绩 |
| 跨职能&Safety | 10% | **3.0** | 业务感强的偏见判断（country × category）；修复偏人工兜底、无 research partnership |

**加权 ≈ 3.00。**

## 两个 3.0 样本的对照（镜像互补）

| 维度 | 样本 A（builder/愿景型） | 样本 B（数据/评测型） |
|------|--------------------------|------------------------|
| 模型 Sense | 2.5（核心概念讲反） | 3.0（概念无硬错） |
| Eval | 3.0（会设计指标） | 3.5（在做 eval 手艺） |
| Product Sense | 3.5（创造力+战略） | 3.0（执行型，缺创造性） |
| Builder | 3.0（GitHub 全绿+真 artifact） | 2.5（纯个人提效+夸大） |
| 数据&商业化 | 2.5 | 3.0（数据域深） |
| 加权 | 2.90 | 3.00 |

**从这对样本提炼的判分先例**：
1. **"会设计指标" ≠ "在做 eval"**：样本 A 把指标列得漂亮 = 3.0；样本 B 的 diff-case→policy 迭代闭环 = 3.5。Eval 给 3.5 看的是手艺，不是指标清单。
2. **Builder 看硬证据**：真 artifact + 可验证使用（A）> 纯个人提效 + 自述夸大（B）。
3. **3.0 档的共性短板**：经典 metrics 题用 engagement/传统视角答（A、B 都栽）——这恰是从 3.0 上 3.5 要补的 AI 视角。

> ⚠️ **采集 caveat**：本样本来自 **pre-v1.1 的较软版本** + 出题**偏简历舒适区**（未攻薄弱处、Product Sense 用的是简历题而非 standalone 硬 case），所以 3.00 **可能略乐观**——真 v1.1 闭卷 + 挑战式追问下，模型 Sense / Product Sense 有下探空间。这条本身就是后来强化"跳出舒适区出题 + 挑战式追问"的由来。
